# Аналітика

Розділ **Analytics** надає вам агрегований вигляд того, що відбувається у вашому застосунку — не лише окремі події, але й тренди, показники помилок і патерни з часом.

## Де знайти

Відкрийте застосунок, потім натисніть іконку **Analytics** в лівій бічній панелі (іконка графіку). Аналітика обчислюється по всіх з'єднаннях для вибраного застосунку.

## Метрики

### Success rate (Успішність)

Відсоток HTTP-запитів, що повернули код статусу 2xx. Здоровий застосунок повинен стабільно тримати понад 99%. Раптові падіння вказують на проблему з деплоєм або бекендом.

### Error rate (Показник помилок)

Відсоток запитів, що повертають 4xx або 5xx. Розбивка за:

- **Client errors (4xx)** — часто вказують на проблеми валідації або автентифікації на стороні клієнта
- **Server errors (5xx)** — вказують на збої бекенду; мають бути близькі до нуля

### Request volume (Обсяг запитів)

Загальна кількість HTTP подій за певний час у вигляді лінійного або стовпчикового графіку. Використовуйте для розуміння патернів навантаження — пікові години, стрибки трафіку після релізу, спади.

### Top failing endpoints (Найпроблемніші endpoint)

Ранжований список endpoint, що генерують найбільше помилок. Це часто найдієвіший вигляд — якщо `/api/checkout` має показник помилок 12%, це ваш перший пріоритет.

```
/api/checkout          12.4%  error rate  (47 errors)
/api/users/profile      3.1%  error rate  (12 errors)
/api/products/search    0.8%  error rate   (3 errors)
```

### Error breakdown by type (Розбивка помилок за типом)

Для JavaScript помилок (не HTTP) — розбивка за типом і повідомленням помилки. Згруповані помилки дають зрозуміти, чи маєте ви 50 екземплярів одного й того самого бага, чи 50 різних.

## Часовий діапазон

Використовуйте селектор часового діапазону для фільтрації аналітики за останню годину, 24 години, 7 днів або довільний діапазон.

## Використання аналітики для налагодження

Поширений workflow:

1. Відкрийте Analytics і помітьте, що `/api/orders` має високий показник помилок з 14:30.
2. Перейдіть до Connections і відфільтруйте за статусом `5xx` та методом `POST`.
3. Знайдіть невдалі запити і розгорніть їх, щоб побачити точну відповідь сервера.
4. Визначте першопричину (наприклад, відсутнє поле бази даних після міграції) і задеплойте виправлення.
5. Спостерігайте, як показник помилок падає до нуля в реальному часі.
